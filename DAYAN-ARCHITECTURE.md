# 大衍筮法 Agent 架构设计

> 底座：《周易·系辞上》"易有太极，是生两仪，两仪生四象，四象生八卦"
> 算法：大衍筮法 — "大衍之数五十，其用四十有九"

---

## 一、宇宙模型：从太极到八卦

```
太极（混沌未分，用户输入前的 Agent 全状态）
 │
 ├─ 两仪（分而为二以象两）
 │   ├─ 阳仪 = Agent 侧（我能做什么）
 │   └─ 阴仪 = 环境侧（任务要什么）
 │
 ├─ 四象（两仪各生阴阳）
 │   ├─ 太阳（阳+阳）= Agent 强 + 环境明 → 直接执行
 │   ├─ 少阴（阳+阴）= Agent 强 + 环境暗 → 谨慎探索
 │   ├─ 少阳（阴+阳）= Agent 弱 + 环境明 → 借力执行
 │   └─ 太阴（阴+阴）= Agent 弱 + 环境暗 → 暂停求助
 │
 └─ 八卦（四象各生阴阳 = 3 爻 × 2 态 = 8 卦）
     └─ 见下方"八卦工具选择"
```

## 二、大衍筮法 → Agent 决策引擎

### 原文对应

| 大衍步骤 | 原文 | Agent 映射 |
|----------|------|-----------|
| 大衍之数五十 | 天地之数，万物之始 | Agent 的全部能力：49 个可用工具 + 1 个"太极常量"（核心循环本身） |
| 其用四十有九 | 去一不用 | 保留 1 个"不动点"：send() 核心循环不可被工具替代 |
| 分而为二以象两 | 随机分为两堆 | 将 49 个工具按"与当前任务的相关性"分为两组：相关 / 不相关 |
| 挂一以象三 | 从左边挂一根于手指 | 从相关组中抽出 1 个"主工具"作为本轮核心执行器 |
| 揲之以四以象四时 | 四根一数 | 对候选工具做四轮评估（见下文"四营"） |
| 归奇于扐以象闰 | 把余数放一边 | 每轮评估的"余数"（不确定性）归入下一轮参考 |
| 五岁再闰 | 闰月需要再扐一次 | 累积不确定性超过阈值时，触发额外一轮评估 |
| 四营而成易 | 四次操作完成一变 | 四轮评估完成，判定一个爻（工具选择结果） |
| 十有八变而成卦 | 18 变成一个六爻卦 | 6 个决策维度 × 3 轮评估 = 18 次计算，得出完整卦象 |

### 核心算法：四营

每"一营"对应一次评估操作，四营完成一爻判定：

```
第一营【分二】：将候选工具分为"直接相关"和"间接相关"两组
    → 余数 r1 = 不相关工具数 % 4（归入环境不确定性）

第二营【挂一】：从直接相关组中选出最佳候选
    → 余数 r2 = 候选与需求的匹配偏差 % 4

第三营【揲四】：对候选做四维评估（能力/成本/风险/历史）
    → 余数 r3 = 评估中的不确定项 % 4

第四营【归奇】：汇总余数，判定爻性
    → r = (r1 + r2 + r3) % 4
    → r=0 → 老阳（old_yang）  → 直接执行，沉淀为高价值路径
    → r=1 → 少阳（young_yang）→ 正常执行
    → r=2 → 少阴（young_yin）→ 需要重试或换方案
    → r=3 → 老阴（old_yin）  → 暂停，请求用户介入
```

### 六爻卦象：六个决策维度

一卦六爻，每爻代表 Agent 决策链中的一个维度：

```
上爻（第六爻）：战略层 — 这个任务值不值得做？
五爻（第五爻）：编排层 — 多个工具/技能的执行顺序
四爻（第四爻）：选择层 — 具体用哪个工具
三爻（第三爻）：参数层 — 工具的参数怎么调
二爻（第二爻）：执行层 — 执行过程中是否需要干预
初爻（第一爻）：反馈层 — 执行结果是否满足预期
```

每个爻经过"三变"（3 轮四营计算）才确定，所以 6 爻 × 3 变 = 18 变。

## 三、数据结构

```python
@dataclass
class YanResult:
    """一爻的判定结果"""
    position: int          # 爻位 (1-6)
    yan_type: str          # 'old_yang' | 'young_yang' | 'young_yin' | 'old_yin'
    remainder: int         # 余数 (0-3)
    changes: list[int]     # 三次变的余数 [r1, r2, r3]
    confidence: float      # 置信度 (0-1)
    reasoning: str         # 判定依据

@dataclass
class GuaResult:
    """完整卦象结果（六爻）"""
    hexagram_name: str     # 卦名（如"天风姤"）
    lines: list[YanResult] # 六爻结果
    inner_trigram: str     # 内卦（下三爻）
    outer_trigram: str     # 外卦（上三爻）
    action_hint: str       # 行动建议
    total_changes: int     # 总变数（应为 18）
    tool_sequence: list    # 推荐的工具执行序列

@dataclass
class DayanState:
    """大衍筮法的中间状态"""
    total_stalks: int = 50           # 大衍之数五十
    active_stalks: int = 49          # 其用四十有九
    reserved_stalk: int = 1          # 去一不用（太极常量）
    separation_a: int = 0            # 左堆（相关工具数）
    separation_b: int = 0            # 右堆（不相关工具数）
    hung_stalk: int = 0              # 挂一（主工具）
    remainders: list[int] = field(default_factory=list)  # 归奇余数
```

## 四、八卦工具选择

八卦由三爻组成，对应工具选择的三个阶段：

```
初爻 = 意图识别（这个任务的本质是什么）
二爻 = 工具匹配（哪个工具最合适）
三爻 = 执行策略（怎么调用最有效）

三爻组合 → 八卦：

☰ 乾（阳阳阳）= 意图清晰 + 工具匹配 + 执行可行
  → 直接执行，全速推进

☱ 兑（阴阳阳）= 意图模糊但工具匹配 + 执行可行
  → 先确认意图再执行

☲ 离（阳阴阳）= 意图清晰 + 工具不匹配 + 执行可行
  → 换工具或万物组合

☳ 震（阳阳阴）= 意图清晰 + 工具匹配 + 执行有风险
  → 谨慎执行，准备回退

☴ 巽（阴阳阴）= 意图模糊 + 工具不匹配 + 执行可行
  → 探索式执行，小步快走

☵ 坎（阳阴阴）= 意图清晰 + 工具不匹配 + 执行有风险
  → 分步执行，降低风险

☶ 艮（阴阳阴）= 意图模糊 + 工具匹配 + 执行有风险
  → 暂停，请求用户澄清

☷ 坤（阴阴阴）= 全部不确定
  → 完全暂停，请求用户介入
```

```
用户输入 "帮我整理桌面"
    │
    ▼
═══ 太极 ═══════════════════════════════════════════
    │
    ▼
═══ 第一变：初爻（反馈层）══════════════════════════
    │ 四营计算：
    │   分二：49 工具 → 相关 12 / 不相关 37
    │   挂一：选出 organize_directory
    │   揲四：能力✓ 成本✓ 风险低✓ 历史✓
    │   归奇：r=1 → 少阳
    │ 判定：少阳 → 正常执行
    │
    ▼
═══ 第二变：二爻（执行层）══════════════════════════
    │ 四营计算：
    │   分二：12 相关工具 → scan_files, organize_directory, ...
    │   挂一：scan_files（先扫描再整理）
    │   揲四：能力✓ 成本低✓ 风险无✓ 历史成功✓
    │   归奇：r=0 → 老阳！
    │ 判定：老阳 → 直接执行，标记高价值
    │
    ▼

    │ ...（以此类推，6 爻 × 3 变 = 18 变）
    │
    ▼
═══ 得卦 ════════════════════════════════════════════
    │

    │ 上三爻 = 外卦（环境状态）
    │ 下三爻 = 内卦（Agent 状态）
    │
    │ 如：☴下☰上 → 天风姤 → execute_with_watch
    │
    ▼
═══ 执行 ════════════════════════════════════════════
    │

    │
    ▼
═══ 归档 ════════════════════════════════════════════
    │
    │ 记录完整卦象 + 执行结果 + 用户反馈
    │ 更新 skill_pairs（五行生克数据）
    │ 更新 time_patterns（时辰规律）
    │ 万物沉淀检查
    │
    ▼
    返回结果
```

## 七、关键设计决策

### Q: 为什么用大衍法而不是梅花易数？

大衍法是"十八变成卦"，过程严谨、可追溯。每一步都有明确的数学操作和余数记录。
梅花易数虽然快捷（报数起卦），但对 Agent 来说太随机，不可复现。

大衍法的"分二"是随机的（类比环境的不确定性），但"揲四"是确定的（类比 Agent 的评估逻辑）。
**随机中有确定，确定中有随机** — 这正是 Agent 需要的决策哲学。

### Q: 18 变会不会太慢？

不会。每"变"是纯计算（查表 + 加法），不需要 LLM 调用。
18 变的总耗时 < 10ms。比现在的 `_taiji_diagnose` 还快。

### Q: 为什么保留"去一不用"？

"其用四十有九"——50 根蓍草去掉 1 根不用。
映射到 Agent：49 个工具 + 1 个"不动点"（send() 核心循环本身）。
这个不动点不能被任何工具替代，它是 Agent 存在的根本。

### Q: 八卦和六十四卦的关系？

八卦 = 三爻 = 工具选择（意图/匹配/执行）
六十四卦 = 六爻 = 完整决策（战略/编排/选择/参数/执行/反馈）

八卦是"选工具"，六十四卦是"做决策"。
不需要显式存储 64 卦——六爻组合自动产生卦象，查表即可。

```sql
-- 大衍筮法日志
CREATE TABLE IF NOT EXISTS dayan_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    session_id TEXT,
    user_input TEXT,
    hexagram_name TEXT NOT NULL,
    inner_trigram TEXT,
    outer_trigram TEXT,
    action_hint TEXT,
    line_1_yan TEXT, line_1_remainder INTEGER,
    line_2_yan TEXT, line_2_remainder INTEGER,
    line_3_yan TEXT, line_3_remainder INTEGER,
    line_4_yan TEXT, line_4_remainder INTEGER,
    line_5_yan TEXT, line_5_remainder INTEGER,
    line_6_yan TEXT, line_6_remainder INTEGER,
    total_changes INTEGER DEFAULT 18,
    tool_sequence_json TEXT,
    execution_success INTEGER,
    elapsed_ms INTEGER DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_dayan_time ON dayan_log(timestamp);
CREATE INDEX IF NOT EXISTS idx_dayan_hexagram ON dayan_log(hexagram_name);
```
